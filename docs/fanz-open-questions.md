# 给 Fanz 的确认清单（保修 + 型号）

> 用途：这些是 CS bot 上线前需要 Fanz 书面确认的点。确认前，相关判定都是
> **PROVISIONAL（暂定）**——bot 对客户永远带"以最新官方政策为准 / 师傅上门确认为准"
> 免责，且照片凭证永不自动批保修。零客户 PII。
> 最后更新：2026-07-11（用 27 张 + 本批 6 张真实经销商发票萃取）

---

## A. 保修期（最要紧——报错品牌 = 报错保修期）

1. **Vioz 全系马达保修是 5 年吗？** 目前我们按 Fanz 10 年 / Vioz 5 年区分。请确认 Vioz
   （Windy MK II、Vetta、CF16、FF565 等子线）马达是否统一 5 年。
2. **Vioz 的 LED 部件（LED 板 / LED 套件）有保修吗？** 官方政策文件里只列了 Fanz 的 LED 条目，
   没看到 Vioz 的。目前 Vioz LED 问题走人工核实、不自动判。请确认 Vioz LED 保修年限与过保收费。
3. **上门服务保修年限**：我们按"2025 年前购买 1 年、2025 起 2 年"。请确认这个分界正确。

## B. 送货单 / 收据能不能当保修凭证？（新，需政策拍板）

4. 真实客户会拿**送货单（Delivery Order）、手写收据、报价单**当保修证明发来，上面有型号和日期
   但不是正式税务发票。
   - 目前做法（方案 B）：bot 照样读取并回显，但加一句"这看起来像送货单/收据，同事会再确认购买凭证"，
     **仍不自动判保修**，交人工。
   - 请 Fanz 确认：保修受理**是否必须**正式税务发票？还是送货单/经销商收据也接受？
     （政策文件已写"授权经销商开具的发票即为有效凭证"，但没明说 DO/收据。）

## C. 型号归属 / 缺失型号（真实发票里出现、我们表里归不出的）

请对以下写法确认 **品牌（Fanz / Vioz）+ 规范型号族**：

5. `FAN123V`（Purple Fox 单，42"，印了 Fanz 品牌）——是什么型号？
6. `DS11MBK` / `DS13MNK`（Value Led 单，56"，混品牌灯饰店单）——这是 Fanz 吊扇型号还是别的品牌？
7. `FSF-10T`（iBath 手写单，标注 TAKEN）——是 FS 系列吗？品牌？
8. `435` 到底是尺寸还是型号代号？（Eco 435L、Inno 435 L 都用 435，我们倾向"型号代号"）
9. `Vios`（手写，疑 Vioz 拼错）——确认是否即 Vioz。

## D. 已从发票实证、请 Fanz 核对无误的结论

- **V605 是 Fanz 主线（马达 10 年），不是 Vioz。** 发票逐字印 FANZ-V605。
- **Vioz 是 Fanz 旗下子线。** 发票逐字印 `FANZ VIOZ ... WINDY-56-MK2`、`FANZ C/F VIOZ WINDY MK II`。
- Fanz 主线族：Gaze / Grande / Aura / Inno / Spinor / V605 / Eco / Axel / FS 系列 / Smart。
- Vioz 子线族：Windy(MK II) / Vetta / CF16 / FF565。

---

### 附：这些确认怎么落地到 bot
- A 类确认后：若某型号归属与发票不符，在 `lib/warranty.js` 的 `MODEL_BRAND_MAP` 手工覆盖（优先级最高）。
- C 类确认后：把新型号加进 `lib/invoice-reader.js` 的 `FAMILIES` 表。
- B 类若 Fanz 要求"仅限税务发票"：把 `index.js` 发票门改回 `inv.isInvoice` 严格判定即可。
